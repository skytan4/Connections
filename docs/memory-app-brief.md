# Memory App Product Brief

Updated: 2026-04-27

## Product Thesis

This app should not position itself as a generic "brain game" product or imply that it can prevent dementia on its own. The stronger product position is:

Help older adults practice memory in ways that feel useful in daily life, learn strategies that improve recall, and stay encouraged through adaptive exercises that are challenging without feeling punishing.

That framing is better aligned with the evidence and with a premium, dignified user experience.

## What You Already Have Right

### 1. Multiple exercise types is a strong idea

Segmenting memory practice into different exercise types makes sense. It is stronger if the segments map to real-world use cases instead of abstract lab-style categories.

Good examples:

- Names and faces
- Dates and appointments
- Shopping lists and errands
- Number sequences and phone numbers
- Story details and conversation recall
- "Remember to do this later" tasks

### 2. A skills area is worth building

A dedicated area that teaches memory strategies and then tests them is one of the best parts of the concept. Research on older adults supports the value of strategy-based memory interventions, especially when the app teaches people how to remember, not just what to tap.

Examples of strategy lessons:

- Chunking
- Association
- Visualization
- Method of loci
- Spaced retrieval
- Implementation intentions
- External memory support habits

### 3. Adaptive difficulty is the right direction

Your instinct to raise difficulty until the user starts to miss and then pull back is sound. Older-adult cognitive training appears to work best when it stays meaningfully challenging rather than fixed and easy.

### 4. Normalizing failure is important

You are right that users need help understanding that missing some items is part of the process. If the app never stretches them, it will feel good but may not be effective.

## Where I Would Challenge The Current Idea

### 1. Do not segment primarily by age

Age alone is too blunt. A 62-year-old and an 82-year-old can have very different needs, but two 74-year-olds can also differ enormously in vision, hearing, speed, confidence, prior education, device comfort, and baseline cognition.

A better model is to personalize by:

- Baseline performance
- Self-reported goals
- Accessibility needs
- Confidence and comfort with technology
- Whether the user is practicing independently or with family/caregiver support

If you want segmentation, use profiles such as:

- Everyday confidence
- Mild challenge
- Guided support
- Care partner mode

That is more respectful and more clinically sensible than age bands.

### 2. Do not make it only "short-term numbers, dates, names"

Those are good categories, but by themselves they are too narrow. The app should also cover:

- Prospective memory: remembering to do something later
- Working memory: holding and manipulating information briefly
- Attention and distraction resistance
- Processing speed
- Delayed recall

If you only train raw item recall, the app may feel repetitive and less useful in daily life.

### 3. Do not market it like a cure or a prevention tool

The evidence on cognitive training is promising but modest and mixed. Some structured interventions improve trained abilities and some everyday functioning, but commercially marketed brain-game claims often run ahead of the science.

Safer positioning:

- Improve memory habits
- Practice recall skills
- Learn evidence-informed memory strategies
- Build confidence for everyday remembering

Avoid strong claims like:

- Prevents dementia
- Reverses memory loss
- Scientifically guarantees sharper memory

### 4. Do not rely on solo, unsupervised repetition alone

Research in healthy older adults suggests training design matters. Purely home-based, repetitive, unsupervised task grinding may be weaker than guided or socially supported formats.

That does not mean the app must become a clinic product. It does mean it should include structure:

- Guided onboarding
- Clear explanations
- Strategy coaching
- Review sessions
- Booster sessions
- Optional partner, family, or coach support

### 5. Be careful how you frame failure

The app should normalize difficulty, but it should not feel like it is trying to make the user fail. For this audience, dignity and confidence matter.

Better framing:

- "This round is supposed to stretch you."
- "Missing a few is how we find the right level."
- "Let's step back one level and practice the strategy again."

Avoid school-like or punitive framing:

- "You failed."
- "Incorrect again."
- "Try harder."

## Research-Backed Product Direction

## Position The App Around Everyday Memory

Instead of making the app feel like a battery of tests, make it feel like practice for everyday life.

Recommended top-level areas:

1. Practice
2. Skills
3. Progress
4. Support

### Practice

The main product surface.

Suggested practice tracks:

- Names and Faces
- Dates and Appointments
- Lists and Errands
- Everyday Numbers
- Story and Conversation Recall
- Remember Later
- Focus and Speed

### Skills

Teach a memory method, let the user try it immediately, then return to that method in later sessions.

Suggested skills curriculum:

1. Chunk information into smaller groups
2. Build vivid associations
3. Turn names into images
4. Use location-based recall
5. Use spaced retrieval
6. Use if-then planning for future tasks
7. Build external memory supports without shame

### Progress

Do not show a single intelligence-like score.

Show:

- What types of recall are improving
- Which strategies are helping
- Where the app is intentionally challenging the user
- Daily-life wins the user logs
- Weekly consistency

The progress area should feel reassuring and specific, not diagnostic.

## Progress Chart Screens

The app should include a dedicated progress screen set that shows progress separately for each memory area.

Recommended areas:

- Names and Faces
- Dates and Appointments
- Lists and Errands
- Everyday Numbers
- Story and Conversation Recall
- Remember Later
- Focus and Speed
- Strategy Skills

The goal is to help users see that progress can be uneven and still meaningful.

That matters because many users may improve in one area before another, and the app should make that feel normal rather than discouraging.

## Recommended Progress Screen Set

### 1. Progress Home

This is the summary screen.

Show:

- Overall weekly practice streak or consistency
- A simple snapshot of each area
- Which areas are improving
- Which area is currently being practiced most
- A short encouraging summary

Possible summary language:

- "Your strongest gains this month are in names and appointments."
- "Focus practice is still building, which is completely normal."
- "Your consistency is helping the app fine-tune your difficulty."

### 2. Area Detail Screen

Each practice area should have its own detail screen.

Example:

- Names and Faces progress
- Dates and Appointments progress
- Remember Later progress

Each detail screen should show:

- Trend over time
- Current difficulty band
- Accuracy trend
- Strategy usage trend when relevant
- Best recent improvement
- Suggested next practice

### 3. Skills Progress Screen

This screen should track strategy learning separately from exercise performance.

Show:

- Which strategies have been introduced
- Which strategies the user applies successfully
- Which strategies need more review
- Booster lessons due soon

This helps reinforce that memory improvement is not only about raw performance. It is also about learning better methods.

### 4. Weekly Reflection Screen

This should feel more human than analytical.

Show:

- Sessions completed this week
- Areas practiced
- One or two clear wins
- One gentle recommendation for next week

Possible language:

- "You were most steady with list recall this week."
- "Short delayed recall is getting easier."
- "Next week, we'll keep strengthening appointments with shorter review sessions."

## Chart Design Principles

- Use one chart per area rather than one crowded dashboard
- Prefer simple line or bar charts
- Use large labels and large tap targets
- Keep axes easy to read
- Avoid clutter, tiny legends, and dense filters
- Use color consistently for each area
- Pair visual trends with short written interpretation
- Allow week, month, and all-time views

Recommended chart types:

- Line chart for progress over time
- Bar chart for weekly completion
- Small milestone markers for strategy unlocks or booster sessions

## What The Charts Should Measure

Do not rely on one raw score.

Each area can draw from a small set of measures:

- Accuracy
- Recall delay handled successfully
- Difficulty level reached
- Stability or consistency
- Strategy use
- Confidence rating if collected

The app can combine these internally, but the user-facing screen should stay simple.

Better user-facing labels:

- Recall strength
- Practice consistency
- Comfortable difficulty
- Strategy use
- Recent improvement

Avoid labels like:

- Cognitive score
- Brain age
- Percentile rank
- Impairment level

## Progress Framing Principles

The charts should support motivation, not self-judgment.

So the UI should:

- Highlight direction, not perfection
- Show small gains clearly
- Normalize uneven progress
- Explain dips as expected
- Emphasize consistency and strategy use

If performance drops, the app should say things like:

- "Today's practice looked a little harder than usual."
- "Energy and focus change from day to day."
- "We'll adjust and keep building from here."

Not:

- "Decline detected"
- "You are getting worse"
- "Below target"

### Support

Optional support features can make the app more effective and more premium:

- Care partner check-in mode
- Practice together mode
- Gentle reminders
- Printable or shareable memory tips
- "What to do if this feels harder lately" guidance

## Recommended Adaptive Difficulty Model

The adaptive system should do more than just increase list length.

Adjust along multiple axes:

- Number of items
- Delay before recall
- Similarity between options
- Presence of distraction
- Switch from recognition to free recall
- Need to apply a strategy before recall

Recommended loop:

1. Quick calibration at onboarding
2. Start just below frustration threshold
3. Increase difficulty after repeated success
4. Step back after repeated misses
5. Reintroduce the challenge after a review round
6. Schedule booster sessions later

This creates "productive difficulty" without the emotional cost of repeated flat failure.

## Onboarding And Calibration

The app should use a hybrid approach:

- Start with a short, gentle calibration
- Continue adapting through normal usage

Do not open with something that feels like a formal cognitive test. For many older adults, that can create anxiety, defensiveness, or early drop-off.

Better approach:

1. Present onboarding as "getting to know your pace"
2. Start with easy early rounds so the user feels immediate success
3. Use those first 3-5 minutes as a hidden baseline
4. Sample a few dimensions:
   - Recall accuracy
   - Delay tolerance
   - Recognition vs free recall
   - Distraction sensitivity
   - Response confidence
5. Raise difficulty quickly but gently during the first week
6. Keep recalibrating over time because performance will vary with fatigue, stress, sleep, and confidence

This gives you the best of both models:

- The app learns enough up front to avoid obvious mismatch
- The user does not feel like they are being judged
- The system keeps improving its estimate based on real behavior

Recommended language:

- "Let's find a comfortable starting point."
- "We'll begin with a few warm-up rounds."
- "The app adjusts as it learns what feels right for you."

Avoid:

- "Assessment"
- "Screening"
- "Testing your memory level"
- "You scored below..."

## Welcoming Onboarding Experience

The first-run experience should feel warm, encouraging, and dignified.

This matters because many users may arrive with some combination of:

- Worry about memory changes
- Fear of being judged
- Skepticism about "brain apps"
- Low confidence with technology

The onboarding should lower emotional resistance before it asks the user to do anything challenging.

Recommended goals for onboarding:

1. Make the user feel safe
2. Explain the purpose in plain language
3. Normalize practice, imperfection, and gradual progress
4. Create one early success moment
5. Begin personalization without pressure

Recommended onboarding flow:

1. Welcome screen
2. Short reassurance about what the app is and is not
3. Pick goals
4. Pick support preferences
5. Gentle warm-up rounds
6. Positive reflection and next-step recommendation

Suggested screen intent:

### Welcome screen

Communicate calm support, not performance pressure.

Possible direction:

- "A calm place to practice memory, one step at a time."

### Reassurance screen

Set expectations early.

Possible direction:

- "This app is for practice, not perfection."
- "Missing a few is part of how we find the right level."
- "Short, steady practice can be more helpful than pushing too hard."

### Goals screen

Let users choose what matters to them.

Examples:

- Remember names more easily
- Keep track of appointments
- Feel more confident day to day
- Practice staying focused
- Build stronger memory habits

### Support preferences

Offer control without making the app feel clinical.

Examples:

- Gentle reminders on or off
- Practice length: 5, 10, or 15 minutes
- Solo practice or practice with a partner
- Larger text and audio support options

### Warm-up rounds

These should feel like a successful start, not an exam.

Rules:

- First round should be easy
- Feedback should be encouraging and specific
- Difficulty should rise subtly
- Copy should reinforce that the app is learning the user's pace

### Reflection screen

Close onboarding with encouragement and clarity.

Possible direction:

- "You are off to a strong start."
- "We will keep adjusting to your pace."
- "Tomorrow's practice will build from what felt comfortable today."

- Keep instructions short
- Never sound patronizing
- Reinforce that challenge is expected and useful

## Contextual Exercise Help

Each exercise should include a clearly visible help affordance placed right where the user is doing the activity.

This is especially important for this audience. If a user wonders:

- "What is this exercise for?"
- "Why am I doing this?"
- "How does this help me?"

they should be able to get the answer without leaving the screen, navigating to another section, or trying to remember where help lives.

## Core Principle

Answers should appear exactly where the question arises.

Do not make users go to:

- A separate help center
- A buried FAQ
- A distant settings screen
- A tutorial they had to remember from onboarding

## What The Help Affordance Should Explain

For every exercise, the contextual help should explain:

1. What this exercise practices
2. Why it matters in everyday life
3. What kind of challenge the user should expect
4. Why it is included in the app

Example structure:

- "What this practices: remembering names after short delays."
- "Why it matters: this helps with everyday social recall."
- "What to expect: some rounds will feel easy, and some are meant to stretch you."
- "Why it's in the app: practicing this skill can support stronger memory habits in daily life."

## Better Than A Generic Info Icon

An `i` icon alone may be too abstract for some users.

A clearer approach is to pair the icon with text such as:

- "Why this exercise?"
- "What is this helping?"
- "Why am I doing this?"

If space is tight, the icon can still be used, but it should be visually obvious, consistently placed, and introduced early so the user learns what it does.

## Recommended Interaction Pattern

- Place the help affordance near the exercise title
- Keep it visible before and during the exercise
- Open a simple sheet, popover, or inline expansion
- Keep the explanation short and readable
- Let the user dismiss it easily and return to the exact same place

This should feel like reassurance, not documentation.

## Copy Principles For Exercise Help

- Use plain language
- Keep it short
- Avoid medical or academic jargon
- Explain the real-life purpose
- Reinforce that challenge is normal
- Never sound corrective or patronizing

Good examples:

- "This exercise helps you practice holding a few details in mind before recalling them."
- "We include this because everyday memory often depends on short delays and small distractions."
- "If this feels difficult, that does not mean you are doing poorly. It helps us find the right level."

Avoid:

- "This module targets episodic encoding deficits."
- "This task measures short-term impairment."
- "Failure indicates reduced working memory capacity."

## Design Requirements For The Help Affordance

- Large enough to tap comfortably
- Strong contrast
- Consistent position across exercises
- Visible label or clearly taught meaning
- Fully accessible with VoiceOver
- Available without losing progress in the exercise

## Suggested Label Territory

Best options:

- Why this exercise
- What this helps
- Why am I doing this

Weaker options:

- Info
- Learn more
- Details

The stronger labels answer the user's real question immediately.

## Liquid Glass Strategy

Liquid Glass should be used to create a premium functional layer, not as decoration across the entire app.

Apple's current guidance is that Liquid Glass works best for controls and navigation layers, not for the main content layer.

That is a very good fit for this product.

Recommended use:

- Navigation bars
- Toolbars
- Tab bars
- Floating primary controls
- Selected control states where the system already applies it naturally

Recommended non-use:

- Large reading cards
- Exercise content panels
- Long-form instructional text surfaces
- Progress content backgrounds

For this audience, important reading surfaces should stay stable and easy to parse. Too much transparency or visual motion in the content layer can work against readability.

## How Liquid Glass Should Feel In This App

The emotional goal is:

- Premium
- Light
- Modern
- Calm

Not:

- Futuristic
- Shiny for its own sake
- Busy
- Floating everywhere

The glass layer should gently separate controls from content while letting the content remain the star.

## Recommended Combination

Best design mix:

- Soft atmospheric content backgrounds
- Standard materials or solid readable surfaces for content cards
- Liquid Glass reserved for chrome and key controls
- Accent color applied sparingly to the most important action

This follows Apple's current guidance well and is also safer for readability.

## Liquid Glass Color Use

By default, Liquid Glass can pick up color from the content behind it.

If you color it directly, do so sparingly.

Best use of color with Liquid Glass:

- Primary call-to-action buttons
- Selected navigation states
- Important confirmation moments

Weaker use:

- Coloring every glass element
- Using multiple accent colors in navigation chrome
- Tinting glass heavily behind dense text

## No Hard-Coded Colors Or Tints

Coloring and tinting should not be hard coded in screens or components.

This should be a firm design-system rule.

All colors, tint behavior, and visual roles should come from centralized theme tokens so the app can:

- Support accessibility adjustments
- Support light and dark appearances cleanly
- Support readability mode changes
- Evolve the visual design without rewriting screens
- Stay consistent across iPhone and iPad
- Potentially support future branded themes or seasonal variants without recoding the UI

## Required Theming Structure

- Backgrounds
- Surfaces
- Elevated surfaces
- Primary text
- Secondary text
- Accent color
- Progress color
- Warning color
- Success color
- Border color
- Chart series colors
- Interactive tint roles

Components should consume semantic roles such as:

- `surfacePrimary`
- `surfaceSecondary`
- `textPrimary`
- `textSecondary`
- `accentPrimary`
- `accentMuted`
- `progressSupport`

not raw color values.

## Tinting Rules

Tinting behavior should also be centralized.

That includes:

- Primary action tint
- Selected state tint

- Progress state tint
- Liquid Glass emphasis tint when used

This prevents one-off visual decisions from drifting across the app.

## Why This Matters

Hard-coded colors create problems fast:

- Accessibility updates become expensive
- Brand adjustments become tedious
- Dark mode becomes fragile
- Progress charts become inconsistent
- Localization and iPad layouts become harder to visually tune

For this app, visual consistency is part of trust. Token-based theming helps preserve that trust.

## Accessibility From The Start

Accessibility should be treated as a core product requirement from the first screen and the first component.

It should not be handled as a cleanup pass near launch.

For this app, accessibility is especially important because the audience may include users with:

- Low vision
- Reduced contrast sensitivity
- Hearing loss
- Slower processing speed
- Reduced dexterity
- Memory-related difficulty navigating or recalling instructions

The product should assume that some users will need extra visual support and simpler interaction patterns from day one.

## Low-Vision And Readability Mode

The app should be designed so it can support an enhanced readability mode without redesigning every screen.

This may be a dedicated in-app mode, or it may mostly rely on system accessibility support plus a few app-level adjustments.

Recommended capabilities:

- Larger text throughout the app
- Heavier font weights where needed for readability
- Higher contrast surfaces and text
- Larger buttons and larger spacing between controls
- Reduced visual clutter
- Stronger selection states
- Optional simplified screens with less simultaneous information

The important architectural point is this:

The design system should make these adjustments easy globally.

That means:

- Shared typography tokens
- Shared spacing tokens
- Shared color and contrast tokens
- Reusable button and card components
- Reusable chart and progress components

## Accessibility Design Requirements

- Support large Dynamic Type without clipping, overlap, or fixed-height failures
- Respect system accessibility settings
- Provide VoiceOver labels and hints for all important controls
- Avoid color-only meaning
- Ensure strong contrast for text, icons, and chart elements
- Keep tap targets comfortably large
- Avoid hidden or complex gestures for core actions
- Avoid timed interactions that disappear before the user is ready
- Let users repeat instructions easily
- Keep help available in place during exercises
- Preserve the user's location and progress when opening help or settings

## Accessibility Testing Standards

Important screens should be considered incomplete until they work well under:

- Large Dynamic Type
- Bold Text
- Increased Contrast
- VoiceOver
- Reduced Motion when applicable
- Different light conditions and color schemes

The app should also be tested with longer reading times and slower task completion assumptions.

## Accessibility Tone

The app should never make accessibility features feel like a special burden or a fallback for failure.

They should feel normal, respectful, and empowering.

Prefer language like:

- Text size
- Easier reading
- Simpler view
- More guidance

Avoid language that implies deficiency or embarrassment.

## Multilingual Support From The Start

The app should be built for multilingual support from the beginning.

This should not be treated as a later translation pass after the UI is already written.

The goal is:

- Every user-facing string lives in localization resources
- Each supported language can have its own language file or catalog entries
- New languages can be added without recoding screens or business logic

## Localization Architecture Requirements

- No hard-coded user-facing strings in views or business logic
- Use centralized localization resources for all display text
- Keep translation resources separate from code
- Organize strings so they are easy to maintain by feature or screen when the catalog grows
- Support language additions without changing layout code for each string

For Apple platforms, current guidance recommends string catalogs in Xcode 15 and later, with localized resources managed separately from app logic.

Exercise definitions should be designed so text content can be localized without changing the exercise engine.

Examples:

- Exercise title key
- Exercise description key
- Help panel key
- Strategy explanation key
- Feedback message key

This is especially important because this app will likely contain a large amount of supportive and instructional copy.

## Design Requirements For Localization

- Layouts must tolerate longer translated strings
- Buttons and cards should expand naturally
- Fixed-width text containers should be avoided where possible
- Important screens should be tested with long strings
- Text should wrap cleanly
- Truncation should be avoided for important instructions
- Charts, menus, and navigation labels must remain readable in translated form

Localization is not only a content system issue. It is also a layout system issue.

## Right-To-Left And Locale Awareness

The app should be built to respect system locale behavior where possible.

That includes:

- Native date formatting
- Native number formatting
- Native calendar and time conventions when relevant
- Support for right-to-left layout if those languages are added later

Even if the first launch languages are left-to-right only, the architecture should not assume English layout forever.

## Translator-Friendly Practices

To make future language additions easier:

- Add clear comments for translators
- Keep strings context-specific
- Avoid ambiguous one-word labels when context matters
- Avoid concatenating short strings to build sentences
- Keep placeholders explicit and well documented
- Use proper plural handling rather than manual string assembly

This reduces translation errors and helps preserve tone across languages.

## Product Tone Across Languages

This app depends heavily on emotional tone.

So localization should preserve:

- Warmth
- Reassurance
- Clarity
- Dignity
- Encouragement

Translations should not sound overly clinical, childish, or machine-generated.

## Recommended Implementation Direction

For this app, the safest long-term approach is:

1. Use localization resources for all user-facing text
2. Keep exercise and help copy keyed by identifiers
3. Use translator comments throughout
4. Support plurals and variable text through localization-aware formatting
5. Test layouts with long translations early

That gives you a system where adding a new language is mostly a content operation, not a screen rewrite.

## Scores And History Storage

The app should store scores and histories from the beginning.

This is necessary for:

- Adaptive difficulty
- Progress charts
- Weekly reflections
- Booster sessions
- Strategy recommendations
- Care partner features if added later

Without stored history, the app cannot feel genuinely personalized or show meaningful long-term progress.

## What The App Should Store

At a minimum, the app should persist:

- Session date and time
- Exercise type
- Exercise area
- Difficulty level used
- Accuracy or completion result
- Delay length or challenge type when relevant
- Strategy used or prompted
- Confidence rating if collected
- Session duration
- Whether help was used
- Notes or daily-life wins if the user logs them

It should also store higher-level summaries such as:

- Best recent performance by area
- Rolling weekly trends
- Current difficulty band by area
- Strategy mastery or familiarity state
- Last practiced date for each area
- Booster or review recommendations due

## Raw History Vs Displayed Progress

The app should store richer internal data than it shows directly.

That means:

- Keep detailed historical records internally
- Show simplified, supportive summaries in the UI
- Avoid overwhelming users with too much raw performance detail

This gives the product flexibility to improve recommendations later without redesigning the storage model.

## Recommended History Views

Stored history should support screens like:

- Area-by-area progress charts
- Recent sessions list
- Weekly reflection summaries
- Skills review history
- Milestones such as "first successful delayed recall at this level"

History should help users see continuity, not just isolated scores.

## Score Design Guidance

Scores should be useful to the system, but presented carefully to the user.

Recommended approach:

- Store numeric performance values internally
- Translate them into reassuring user-facing progress language
- Avoid a single harsh master score that feels diagnostic

Better user-facing framing:

- Practice strength
- Current comfort level
- Recent improvement
- Strategy progress
- Consistency

Weaker framing:

- Memory score
- Brain score
- Cognitive rating
- Mental age

## Persistence And Trust

Because this app touches sensitive personal experience, storage design should also support trust.

Recommended principles:

- Keep data handling simple and explainable
- Make history feel private and respectful
- Let users understand what is being saved
- Avoid storing unnecessary sensitive information
- Give users clear control over deleting history if needed

If possible, privacy-preserving storage can be a product advantage.

## Product Requirement

Scores and histories are not optional support features.

They are foundational product infrastructure for:

- Personalization
- Motivation
- Reflection
- Retention
- Long-term usefulness

## Privacy And Local-Only Data

This app should make its privacy posture extremely clear:

- User data stays on the device
- The app does not send exercise results, scores, history, or notes to a server
- The app does not require an account
- The app does not rely on tracking or analytics frameworks

For this audience, privacy is not just a compliance issue. It is part of the product's emotional safety.

Users should never have to wonder:

- "Who can see my scores?"
- "Is this being uploaded somewhere?"
- "Will my family or an outside company get this information?"

## Privacy Messaging Requirements

The app should clearly explain, in plain language:

1. What data is stored
2. Where it is stored
3. That it remains local on the device
4. That the developer does not receive it
5. How the user can remove it

This explanation should appear in more than one place:

- Privacy policy
- Settings or About area inside the app
- Potentially a short reassurance note during onboarding

## Recommended Plain-Language Promise

Best core message:

"Your memory practice data stays on your device."

Supporting language:

- "Scores, progress, and history are stored locally on your device."
- "We do not collect or upload your exercise results."
- "We do not sell, share, or track your personal memory data."
- "If you delete the app, your local app data is removed from the device."

## Product Note

If any future version adds cloud sync, backup, account features, research sharing, or analytics, the privacy promise and policy will need to change accordingly.

Until then, the product should be designed to fully support the simpler and stronger local-only promise.

## Copy Direction

The app should say "practice" more often than "test."

Examples:

- "Today's practice"
- "Let's warm up"
- "This one is meant to stretch your memory"
- "You remembered more when you used grouping"
- "We'll revisit this skill later"

Avoid:

- "Brain age"
- "Cognitive score"
- "Pass/fail"
- "You performed below average"

## Product Strategy Recommendations

## Stronger Positioning

Best positioning:

"A guided memory practice app for older adults that teaches real strategies and adapts to your level."

Weaker positioning:

"A brain game app for seniors."

The first sounds premium and trustworthy. The second sounds generic and easier to dismiss.

## Free Vs Paid

If you eventually monetize this, keep the free version genuinely useful.

Free could include:

- Baseline assessment
- Daily practice in one or two tracks
- Intro skills lessons
- Weekly summary

Paid could include:

- Full practice library
- Full skills curriculum
- Personalization plans
- Booster programs
- Care partner features
- Deeper progress insights

Do not put a paywall before the user understands the product.

## Recommended Tagline Territory

Possible directions:

- Practice memory with calm, guided support
- Learn memory strategies that fit everyday life
- Stay sharp with adaptive recall practice
- Build memory confidence, one short session at a time

## Bottom Line

The concept is strong if it becomes a guided, adaptive memory-practice product grounded in daily life and strategy teaching.

The concept becomes weak if it turns into a pile of disconnected mini-games sorted only by age and difficulty.

The biggest recommendation is this:

Build around everyday memory goals and strategy learning, not around abstract "brain training" claims.
